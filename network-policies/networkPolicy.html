What are Network Policies?

By default, Kubernetes allows all pods to communicate with all other pods across the cluster — no restrictions. A Network Policy is a Kubernetes resource that acts like a firewall ruleset at the pod level, controlling which pods (or namespaces, or external IPs) can send traffic to or from specific pods.

Analogy Think of your cluster as an office building. By default, every door is unlocked and everyone can walk anywhere. A NetworkPolicy is your keycard access system — it locks everything, then explicitly says "the frontend team can enter the backend room."

Why They Matter in Real-World Clusters

🔒 Security Isolation

If a pod is compromised, an attacker can't laterally move to your database or internal services without explicitly allowed paths.

🎯 Zero-Trust Networking

Never trust, always verify. Every pod must earn access. Default-deny-all then explicitly allow needed traffic.

🏢 Multi-Tenant Compliance

In shared clusters (HIPAA, PCI-DSS), isolate tenant namespaces so workloads can never bleed across boundaries.

🐛 Blast Radius Reduction

A bug in your payment service can't accidentally call your auth service if no network policy permits it.

How They Work — The Building Blocks

A NetworkPolicy selects pods via podSelector, then defines ingress (incoming) and/or egress (outgoing) rules. Each rule can filter by:

Selector TypeWhat It TargetsExample Use Case
podSelectorPods within the same namespace by labelsAllow only app=frontend to reach app=backend
namespaceSelectorAll pods from a specific namespaceAllow monitoring namespace to scrape metrics
ipBlock (CIDR)External IP rangesAllow only company VPN IP range (10.0.0.0/8)
portsSpecific port + protocolOnly allow TCP:5432 for PostgreSQL connections

Anatomy of a Network Policy YAML

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: ecommerce         # scope: this namespace only

spec:
  # 1. Which pods does this policy APPLY TO? (the "target")
  podSelector:
    matchLabels:
      app: backend              # this policy governs backend pods

  # 2. Which traffic directions are we controlling?
  policyTypes:
    - Ingress                  # we're controlling INCOMING traffic

  # 3. Define WHAT is allowed in
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend     # ONLY pods labeled app=frontend
      ports:
        - protocol: TCP
          port: 8080            # only on port 8080
⚠ CNI Plugin Requirement Network Policies are just Kubernetes API objects — they do nothing without a CNI plugin that enforces them. Vanilla cluster CNIs like flannel do NOT enforce policies. You need Calico, Cilium, Weave, or Antrea. Most managed clusters (EKS, GKE, AKS) have this pre-configured.

Core Concepts Breakdown

The mechanics behind ingress/egress rules, default behaviors, label selection, and CNI plugins.

Ingress vs. Egress

⬇ Ingress (Incoming Traffic)

Controls WHO can send traffic TO the selected pods. Think: "Who is allowed to knock on my door?"

Use case: Allow only frontend pods to call the backend API on port 8080.

⬆ Egress (Outgoing Traffic)

Controls WHERE the selected pods can send traffic. Think: "Which doors can my pod walk through?"

Use case: Prevent backend pods from calling anything except the DB on port 5432.

Key mental model If you declare policyTypes: [Ingress] on a policy, that policy controls only incoming traffic to the selected pods. Any ingress NOT matching an allow rule is dropped. But egress is still wide open unless you also write an Egress policy (or declare both in one policy).

Default Allow vs Default Deny

This is the most important concept to internalize early:

ScenarioBehavior
No NetworkPolicy selects a podAll ingress AND egress allowed (open by default)
A NetworkPolicy selects a pod (Ingress)ALL ingress blocked except rules in that policy. Egress still open.
A NetworkPolicy selects a pod (Egress)ALL egress blocked except rules in that policy. Ingress still open.
Both Ingress + Egress policies select a podBoth directions are locked down — pure whitelist mode

The Default-Deny-All Pattern (Zero Trust Baseline)

# Step 1: Lock everything down
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: ecommerce
spec:
  podSelector: {}    # empty = matches ALL pods in namespace
  policyTypes:
    - Ingress
    - Egress
  # no ingress/egress rules = deny everything

After applying this, all pods in the namespace are completely isolated. Then you add individual allow-rules one by one.

Label-Based Selection (The Heart of Network Policies)

Network Policies are entirely driven by pod labels. This is why label hygiene is critical in production.

Single Label Match

podSelector:
  matchLabels:
    app: backend         # exact match: app must equal "backend"

Multi-Label AND Match

podSelector:
  matchLabels:
    app: backend
    tier: api            # BOTH labels must match (AND logic)

In Ingress Rules: AND vs OR between from[] items

# CASE 1: This is an AND (same list item) — BOTH conditions must be true
# (Pod labeled app=frontend) AND (from namespace labeled env=prod)
ingress:
  - from:
      - namespaceSelector:
          matchLabels:
            env: prod
        podSelector:      # same dash level = AND
          matchLabels:
            app: frontend

---

# CASE 2: This is an OR (separate list items) — EITHER condition is enough
# (Any pod in namespace env=prod) OR (Any pod labeled app=frontend)
ingress:
  - from:
      - namespaceSelector:  # separate dash = OR
          matchLabels:
            env: prod
      - podSelector:        # separate dash = OR
          matchLabels:
            app: frontend
🚨 Most common beginner mistake! The AND vs OR distinction (same dash vs separate dash in the from[] array) is the #1 source of confusion. Draw it out when writing complex rules.

3
Egress: Restrict Backend to Only Talk to the Database
Goal: The backend pod can only send traffic OUT to PostgreSQL on port 5432
Intermediate
Important: Remember to also allow egress to kube-dns (UDP/TCP port 53) — otherwise your pod can't resolve service names!
# ex3-backend-egress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-egress-only-db
  namespace: netpol-demo
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Egress
  egress:
    # Rule 1: Allow DNS lookups (critical — don't forget this!)
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53

    # Rule 2: Allow connection to database on port 5432
    - to:
        - podSelector:
            matchLabels:
              app: postgres
      ports:
        - protocol: TCP
          port: 5432

Test

kubectl exec backend -n netpol-demo -- nc -zv <postgres-ip> 5432
kubectl exec backend -n netpol-demo -- nc -zv google.com 443
Expected: DB connection succeeds. Internet access (google.com) is blocked. Backend is now "egress jailed."
4
Namespace Isolation — Prevent Cross-Namespace Leakage
Goal: Pods in namespace "team-a" cannot communicate with pods in "team-b"
Intermediate 
# Setup two namespaces
kubectl create namespace team-a
kubectl create namespace team-b
kubectl label namespace team-a team=a
kubectl label namespace team-b team=b
# ex4-namespace-isolation.yaml
# Apply this in BOTH namespaces (change namespace field for each)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-other-namespaces
  namespace: team-a              # also apply to team-b
spec:
  podSelector: {}                 # all pods in this namespace
  policyTypes:
    - Ingress
  ingress:
    - from:
        # Only allow ingress from pods within the SAME namespace
        - podSelector: {}          # {} = all pods
          # Note: namespaceSelector omitted = same namespace only
Why no namespaceSelector? When only podSelector is present in a from entry, it matches pods in the same namespace as the policy. This elegantly creates namespace isolation.

Test

kubectl run pod-a --image=busybox:1.28 -n team-a -- sleep 3600
kubectl run pod-b --image=busybox:1.28 -n team-b -- sleep 3600
kubectl exec pod-a -n team-a -- wget -qO- http://<pod-b-ip> --timeout=3
Expected: Cross-namespace connection from team-a to team-b is blocked.
5
CIDR-Based External Traffic Control
Goal: Allow a backend pod to reach an external payment API but block all other internet
Intermediate 
# ex5-cidr-egress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-payment-api
  namespace: netpol-demo
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Egress
  egress:
    # DNS (always needed)
    - ports:
        - port: 53
          protocol: UDP
        - port: 53
          protocol: TCP

    # Allow HTTPS to Stripe payment API IP range (example)
    - to:
        - ipBlock:
            cidr: 54.187.174.0/24    # Stripe IP range (example)
            except:
              - 54.187.174.64/30    # exclude a specific sub-range
      ports:
        - protocol: TCP
          port: 443

    # Allow internal cluster traffic
    - to:
        - ipBlock:
            cidr: 10.0.0.0/8         # internal cluster CIDRs
Real-world tip: Payment providers like Stripe, PayPal publish their IP ranges. Always check their docs for official CIDRs to use in ipBlock.
6
Ingress Controller Gateway Pattern (Advanced)
Goal: Only the Nginx ingress controller can talk to the frontend. Block all direct pod access.
Advanced

This pattern ensures all external traffic flows through the ingress controller — a real production requirement.

# First, label your ingress-nginx namespace
kubectl label namespace ingress-nginx kubernetes.io/metadata.name=ingress-nginx
# ex6-ingress-controller-only.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-controller-only
  namespace: ecommerce
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
    - Ingress
  ingress:
    # Allow ONLY from the ingress-nginx namespace
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
          podSelector:
            matchLabels:
              app.kubernetes.io/name: ingress-nginx  # AND from the ingress pods
      ports:
        - protocol: TCP
          port: 3000              # your frontend's service port

Test end-to-end

curl -H "Host: myapp.example.com" http://<ingress-ip>/
kubectl exec busybox -n ecommerce -- wget -qO- http://<frontend-clusterip>:3000 --timeout=3
Expected: curl via ingress works (traffic flows correctly). Direct pod-to-frontend wget is blocked — external access is forced through the ingress controller.

Debugging & Troubleshooting

Common mistakes, verification strategies, and the kubectl commands you'll use daily.

Common Beginner Mistakes

❌ Mistake 1: Forgetting DNS Egress

You lock down egress but forget to allow port 53. Your pod can't resolve postgres-service to an IP and silently fails.

Fix: Always add a DNS egress rule to every egress policy: allow UDP+TCP port 53 to kube-system or allow it globally on port 53.
❌ Mistake 2: AND vs OR Confusion in from[]

Putting namespaceSelector and podSelector as separate items when you meant the combination — accidentally allowing entire namespaces.

# WRONG — this allows ALL pods in prod-ns OR all pods named frontend
- from:
    - namespaceSelector:
        matchLabels:
          env: prod
    - podSelector:
        matchLabels:
          app: frontend

# CORRECT — only frontend pods from prod namespace (AND)
- from:
    - namespaceSelector:
        matchLabels:
          env: prod
      podSelector:
        matchLabels:
          app: frontend
❌ Mistake 3: Policies on Pods That Don't Have Matching Labels

A policy targeting app=backend does nothing if your pods are labeled app=backend-service (even a typo breaks it). Always double-check label alignment.

❌ Mistake 4: Using flannel CNI (No Enforcement)

Your policies exist in etcd, Kubernetes accepts them — but flannel silently ignores them. All traffic flows as if the policy doesn't exist. This is dangerous because you think you're protected.

Fix: Verify your CNI: kubectl get pods -n kube-system | grep cni. If it's flannel, switch to Calico or Cilium.
❌ Mistake 5: Not Labeling Namespaces

Using namespaceSelector but forgetting to add labels to the target namespace. The policy won't match anything.

# Always label namespaces you plan to use in policies
kubectl label namespace monitoring team=platform env=prod
kubectl label namespace ecommerce app=ecommerce env=prod

Essential Debugging Commands

  • kubectl get networkpolicy -n <namespace> -o wide
    List all policies in a namespace with their pod selectors
  • kubectl describe networkpolicy <name> -n <namespace>
    Full details of a policy — shows resolved rules and pod selector matches
  • kubectl get pods -n <ns> --show-labels
    Verify pod labels match what your policy is targeting
  • kubectl exec <pod> -n <ns> -- nc -zv <target-ip> <port>
    Test TCP connectivity from inside a pod. -z = zero-I/O (port scan), -v = verbose
  • kubectl run debug --image=nicolaka/netshoot --rm -it -n <ns> -- bash
    Launch an ephemeral debug pod with full networking tools (curl, nc, dig, nmap, tcpdump)
  • kubectl exec <pod> -- nslookup kubernetes.default
    Test DNS resolution is working. If this fails, your egress DNS policy is broken
  • kubectl get pods -n kube-system -l k8s-app=calico-node
    Verify Calico agent pods are running on all nodes
  • calicoctl get networkpolicy --all-namespaces -o yaml
    View policies from Calico's perspective (may differ from kubectl view)
  • kubectl get events -n <ns> --sort-by='.lastTimestamp'
    Check for any policy-related warning events

Systematic Debugging Workflow

  • 1.Check if a NetworkPolicy exists: kubectl get netpol -n <ns>. If none exist but traffic is blocked — look at your CNI configuration.
  • 2.Verify pod labels match: kubectl get pod <name> -o jsonpath='{.metadata.labels}'. Compare to policy's podSelector.
  • 3.Check namespace labels: kubectl get ns <name> --show-labels. Cross-namespace policies require correct namespace labels.
  • 4.Test connectivity with netshoot: nc -zv <ip> <port> from both source and destination.
  • 5.Check CNI logs: kubectl logs -n kube-system -l k8s-app=calico-node --tail=50 for policy application errors.
  • 6.Temporarily remove the policy and re-test. If traffic flows — the policy is blocking. Re-add and refine rules.

Best Practices

Production-level recommendations for secure, maintainable Network Policy configurations.

The Zero-Trust Policy Template

For every namespace you create in production, apply this pattern immediately — before deploying any workloads:

  1. Apply default-deny-all (blocks all ingress and egress)
  2. Allow DNS egress (UDP/TCP 53 to kube-system)
  3. Allow health check probes if needed (kubelet → pod)
  4. Explicitly allow each required service-to-service path
  5. Allow monitoring namespace to scrape metrics

Naming & Labeling Strategy

Policy Naming Convention

# Pattern: <direction>-<source/target>-<what>
# or:      <service>-<direction>-<description>

default-deny-all              # baseline lockdown
frontend-egress-to-backend    # who sends where
backend-ingress-from-frontend # who receives from whom
postgres-allow-backend-only   # guard for database
allow-monitoring-scrape       # cross-namespace access
allow-ingress-controller      # external entry point

Recommended Pod Label Schema

labels:
  app: backend           # service name (used in policies)
  tier: api              # architectural tier: web/api/database/cache
  component: rest        # implementation type
  version: v2            # useful for canary policies
  part-of: ecommerce     # parent application
  managed-by: helm       # deployment tool
Never use unstable labels in policies. Labels like pod-template-hash or release-specific labels change on every deploy. Only use stable, intent-based labels like app= and tier= in your NetworkPolicy selectors.

Production Security Tips

  • Start with default-deny-all in every namespace — never add policies to an open namespace
  • Use policyTypes: [Ingress, Egress] explicitly — don't rely on implicit behavior
  • Always test policies in a staging namespace before applying to production
  • Never allow port 0 (all ports) unless absolutely necessary
  • Separate the database into its own namespace for stronger isolation boundaries
  • Use Cilium's NetworkPolicy Editor (editor.cilium.io) to visually design and validate policies
  • Store all NetworkPolicy YAMLs in Git — treat them as Infrastructure as Code
  • Use Kyverno or OPA/Gatekeeper to enforce that every new namespace gets a default-deny policy
  • Regularly audit: kubectl get netpol --all-namespaces to ensure no namespaces are unprotected
  • Don't disable NetworkPolicies "temporarily" in production — create narrow allow-rules instead
  • Don't use podSelector: {} in ingress rules (allows all pods) without understanding the blast radius

Advanced Topics

Namespace isolation patterns, DNS policies, service mesh integration, and CNI-specific extensions.

Advanced Namespace Isolation

For large organizations or multi-tenant clusters, per-namespace isolation is crucial. This pattern creates a "namespace jail":

# Apply once per namespace — creates a hard boundary
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: namespace-isolation
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
  ingress:
    # Allow from same namespace
    - from:
        - podSelector: {}
    # Allow from monitoring namespace
    - from:
        - namespaceSelector:
            matchLabels:
              purpose: monitoring
  egress:
    # DNS
    - ports:
        - port: 53
          protocol: UDP
    # Allow to same namespace only
    - to:
        - podSelector: {}

DNS Policies

Kubernetes' internal DNS (CoreDNS) runs in kube-system. Egress policies that lock down DNS break service discovery. There are three approaches:

Approach 1: Port 53 to kube-system
egress:
  - to:
      - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: kube-system
    ports:
      - port: 53
        protocol: UDP

✓ Most specific, recommended

Approach 2: Port 53 everywhere
egress:
  - ports:
      - port: 53
        protocol: UDP
      - port: 53
        protocol: TCP

⚠ Simpler but less restrictive

Pod DNS policy (dnsPolicy field) is separate from NetworkPolicy — dnsPolicy: ClusterFirst (the default) means the pod uses CoreDNS. dnsPolicy: None lets you specify custom DNS servers. This is a pod-level setting, not network-level.

Calico: GlobalNetworkPolicy

Calico extends Kubernetes NetworkPolicy with a cluster-wide GlobalNetworkPolicy that spans all namespaces — useful for organization-wide security baseline rules.

# Calico-specific resource — not native Kubernetes
apiVersion: projectcalico.org/v3
kind: GlobalNetworkPolicy
metadata:
  name: deny-all-external-egress
spec:
  order: 1000               # lower number = higher priority
  selector: all()           # applies to ALL pods in ALL namespaces
  types:
    - Egress
  egress:
    # Allow internal cluster traffic
    - action: Allow
      destination:
        nets:
          - 10.0.0.0/8
    # Deny everything else (internet)
    - action: Deny

Cilium: Layer 7 HTTP Policies

Cilium's eBPF-based implementation can enforce policies at the HTTP/gRPC level — beyond IP/port to actual HTTP paths and methods.

# Cilium NetworkPolicy with L7 HTTP rules
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-only-get-products
spec:
  endpointSelector:
    matchLabels:
      app: backend
  ingress:
    - fromEndpoints:
        - matchLabels:
            app: frontend
      toPorts:
        - ports:
            - port: "8080"
              protocol: TCP
          rules:
            http:
              # Only allow GET /api/products — block all other paths/methods
              - method: GET
                path: /api/products
              - method: GET
                path: /api/products/.*  # regex supported
This is powerful — Cilium can block POST /api/admin even from the frontend pod. Standard NetworkPolicy can only control at IP/port level. This is L7 micro-segmentation.

Service Mesh Integration (Istio + NetworkPolicy)

A service mesh like Istio operates at the application layer (L7) and handles mTLS, retries, and traffic shaping. It complements (but doesn't replace) NetworkPolicy:

FeatureNetworkPolicy (L3/L4)Istio AuthorizationPolicy (L7)
IP/Port filtering✅ Yes⚠ Partial
HTTP path/method filtering❌ No✅ Yes
mTLS encryption❌ No✅ Yes
Header-based routing❌ No✅ Yes
Enforced byCNI plugin (kernel)Envoy sidecar (app layer)
OverheadMinimalHigher (sidecar per pod)
Best practice with Istio: Still use NetworkPolicy as a defence-in-depth layer. Lock down IP/ports with NetworkPolicy, then use Istio's AuthorizationPolicy for fine-grained L7 control like JWT validation, HTTP path matching, and service identity-based authorization. 
# Istio AuthorizationPolicy — works alongside NetworkPolicy
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: backend-authz
  namespace: ecommerce
spec:
  selector:
    matchLabels:
      app: backend
  action: ALLOW
  rules:
    - from:
        - source:
            # Enforce mTLS identity — not just label-based
            principals:
              - cluster.local/ns/ecommerce/sa/frontend
      to:
        - operation:
            methods: ["GET", "POST"]
            paths: ["/api/*"]
